# 包功能与使用说明

## 概述

`MIPTD` 的全称是 `Multi-source Integrative Progressive Target Discovery`。

这个包的目标是把单个 `CAS` 分子的 Figure 1 分析流程做成一个可重复、可命令行执行、以单分子项目为单位组织结果的 Python 包。

设计原则只有三条：

- 单个 `CAS` 对应一个独立项目目录
- 默认在线运行，不依赖历史 demo 结果
- 结果按 `a -> b -> c -> d -> e` 顺序组织，并且每一步都可追踪

## 总流程图

下面这张总流程图概括了当前包的主执行链：

```text
输入 CAS
  ->
PubChem 在线解析
  ->
得到 compound / name / formula / canonical_smiles / inchikey
  ->
Step A: 四库靶标收集
  ->
  SwissTargetPrediction    <- 使用 canonical_smiles
  SEA                      <- 使用 canonical_smiles
  PPB2                     <- 使用 canonical_smiles
  ChEMBL                   <- 使用 name + inchikey + canonical_smiles
  ->
统一到人类基因层级
  ->
构建 venn_inputs.json / venn.rds / source_summary.json
  ->
Step B: 四库人类靶标 Venn 图
  ->
Step C: 四库靶标并集 GO / KEGG 富集
  ->
疾病关键词过滤并绘制 c 图
  ->
Step D: 提取关键 KEGG 基因并绘制 circlize 和弦图
  ->
Step E: 从 d 图关键基因构建 Chemprop multi-task
  ->
  候选靶标映射
  ->
  ChEMBL 人类活性数据整理
  ->
  训练模型
  ->
  评估模型
  ->
  预测当前单个 CAS 分子
  ->
  过滤低质量任务
  ->
输出 Figure 1e 与最终靶标排序结果
```

如果按 Figure 1 panel 对应关系看，流程可以进一步简化为：

```text
CAS
  ->
PubChem
  ->
Swiss / SEA / ChEMBL / PPB2
  ->
a: 数据来源与流程示意
  ->
b: 四库人类靶标 Venn
  ->
c: 并集靶标 GO / KEGG + 疾病关键词过滤
  ->
d: 关键 KEGG 基因 circlize
  ->
e: Chemprop multi-task target prioritization
```

这个包面向的核心任务是：

1. 输入一个真实 `CAS`
2. 在线解析化合物身份信息
3. 在线从四个来源抓取人类靶标
4. 完成 Figure 1 的 `a -> e` 全流程
5. 在当前项目目录下生成结构化结果

## 包能做什么

当前包已经具备以下能力：

- 输入 `CAS` 后在线解析化合物名称、分子式、`SMILES`、`InChIKey`
- 使用解析出的 `SMILES` 提交到：
  - `SwissTargetPrediction`
  - `SEA`
  - `PPB2`
- 使用解析出的名称、`InChIKey`、`SMILES` 匹配 `ChEMBL`
- 整合四库结果并构建 `Venn`
- 对并集靶标做 `GO/KEGG` 富集分析
- 基于肝脏疾病关键词筛选通路并绘制 `c` 图
- 基于关键 KEGG 基因绘制 `d` 图
- 严格从 `d` 图关键基因出发构建 `Chemprop` 任务并生成 `e` 图
- 输出模型测试指标、过滤依据和过滤后的靶标结论
- 对最终 case 目录做单 `CAS` 一致性校验

## 分析流程说明

### Step 0：输入与在线身份解析

包的真正入口参数是：

- `CAS`
- 疾病关键词列表

系统首先会在线解析化合物身份信息，并保存成正式产物：

- `compound_identity.json`
- `compound_identity.csv`

保存位置有两个：

- case 根目录
- `a_source_collection/`

目前解析来源是 `PubChem PUG REST`。

输出字段包括：

- `compound`
- `cas`
- `name`
- `formula`
- `canonical_smiles`
- `inchikey`
- `identity_source`
- `pubchem_cid`

此外，当前包内置了一份正式的人类映射资源：

- [`src/miptd/resources/idmapping.tsv`](/Users/liuyue/Desktop/workspace/Cadisum_MM_Project/Nature_Communication_Figure1/src/miptd/resources/idmapping.tsv)

这份文件用于把 `UniProt accession` 映射成标准基因符号，支撑 `Swiss / ChEMBL / PPB2` 的基因层级统一。

同时，当前包也内置了一份正式的人类 `ChEMBL target catalog`：

- [`src/miptd/resources/ChEMBL_target_catalog.csv`](/Users/liuyue/Desktop/workspace/Cadisum_MM_Project/Nature_Communication_Figure1/src/miptd/resources/ChEMBL_target_catalog.csv)

这份文件用于把 `ChEMBL target ID` 解析为 `Accessions / Type / Organism`，支撑 `PPB2` 标准化以及 `Chemprop` 阶段的可训练靶标映射。

如果需要重建这两份正式资源，可直接运行：

```bash
Rscript scripts/build_idmapping_tsv.R --output src/miptd/resources/idmapping.tsv
python3 scripts/build_chembl_target_catalog.py \
  --output-csv src/miptd/resources/ChEMBL_target_catalog.csv \
  --summary-json src/miptd/resources/ChEMBL_target_catalog.summary.json
```

### 四个数据库抓取说明

下面四个来源都对应独立抓取脚本，统一放在 [`scripts`](/Users/liuyue/Desktop/workspace/Cadisum_MM_Project/Nature_Communication_Figure1/scripts) 下。

#### SwissTargetPrediction

对应脚本：

- [`scripts/fetch_swiss_targets.py`](/Users/liuyue/Desktop/workspace/Cadisum_MM_Project/Nature_Communication_Figure1/scripts/fetch_swiss_targets.py)

抓取方式：

- 先输入解析得到的 `SMILES`
- 使用 `Playwright + Chrome` 打开 `SwissTargetPrediction` 网站
- 在真实网页中填写 `SMILES`
- 选择 `Homo sapiens`
- 提交任务并轮询结果页
- 解析结果表格并输出为标准 CSV

主要特点：

- 这是四个来源里最接近真实浏览器行为的一条线
- 结果直接来自 Swiss 的结果页表格
- 输出文件通常是 `SwissTargetPrediction_<CAS>.csv`

#### SEA

对应脚本：

- [`scripts/fetch_sea_targets.py`](/Users/liuyue/Desktop/workspace/Cadisum_MM_Project/Nature_Communication_Figure1/scripts/fetch_sea_targets.py)

抓取方式：

- 先输入解析得到的 `SMILES`
- 使用 `requests.Session()` 打开 SEA 首页
- 读取页面中的 `csrf_token`
- 以表单方式提交 `SMILES`
- 轮询结果页直到任务完成
- 解析结果表格
- 只保留 `Target ID` 以 `_HUMAN` 结尾的人类结果

主要特点：

- 不是完整浏览器自动化，而是浏览器风格会话请求
- 通过 `csrf_token + session` 维持在线查询流程
- 输出文件通常是 `sea-results.xls`

#### ChEMBL

对应脚本：

- [`scripts/fetch_chembl_targets.py`](/Users/liuyue/Desktop/workspace/Cadisum_MM_Project/Nature_Communication_Figure1/scripts/fetch_chembl_targets.py)

抓取方式：

- 输入化合物名称，并结合 `InChIKey` 与 `SMILES` 做精确匹配
- 先调用 `ChEMBL molecule search API` 定位分子
- 再拉取该分子的全部 `activity` 记录
- 只保留 `target_organism == "Homo sapiens"` 的活性
- 统计对应的 `target_chembl_id`
- 再逐个拉取 target 元信息，生成标准化人类靶点表

主要特点：

- 这一条使用的是 `ChEMBL` 公共 API，不依赖网页模拟
- 它的分析链是：
  `compound -> ChEMBL molecule -> activity -> human target`
- 输出文件通常是 `chembl_targets.csv`

#### PPB2

对应脚本：

- [`scripts/fetch_ppb2_targets.py`](/Users/liuyue/Desktop/workspace/Cadisum_MM_Project/Nature_Communication_Figure1/scripts/fetch_ppb2_targets.py)

抓取方式：

- 先输入解析得到的 `SMILES`
- 脚本把 `SMILES` 作为 `smi` 参数拼进 PPB2 的预测 URL
- 再附带：
  - `fp`
  - `method`
  - `scoringmethod`
- 直接请求 PPB2 结果页面
- 解析返回 HTML 中的 target 结果表
- 再结合：
  - [`src/miptd/resources/ChEMBL_target_catalog.csv`](/Users/liuyue/Desktop/workspace/Cadisum_MM_Project/Nature_Communication_Figure1/src/miptd/resources/ChEMBL_target_catalog.csv)
  - [`src/miptd/resources/idmapping.tsv`](/Users/liuyue/Desktop/workspace/Cadisum_MM_Project/Nature_Communication_Figure1/src/miptd/resources/idmapping.tsv)
  把 `ChEMBL target ID` 映射成人类基因符号

主要特点：

- 这条线不是往网页输入框里手工填值
- 而是直接构造 PPB2 后端接受的预测 URL
- 输出文件通常是 `ppb2_targets.csv`

### Step A：四库靶标收集

这一阶段使用在线解析出的身份信息驱动四个来源：

- `SwissTargetPrediction`
- `SEA`
- `ChEMBL`
- `PPB2`

这里有一个关键规则：

- `CAS` 只是入口键
- `Swiss/SEA/PPB2` 实际提交的是解析后的 `SMILES`
- `ChEMBL` 匹配用的是名称、`InChIKey` 和 `SMILES`

本步骤的主要结果包括：

- 四库原始结果文件
- `venn_inputs.json`
- `venn.rds`
- `source_summary.json`

### Step B：Venn

四个来源的结果会统一到同一套基因层级后，构建 `Venn`。

输出到：

- `b_venn/1.p_venn.pdf`

### Step C：GO/KEGG

四库靶标取并集后做：

- `GO enrichment`
- `KEGG enrichment`

然后基于疾病关键词做通路筛选，并生成 `c` 图。

这里的“过滤”逻辑需要明确：

1. 先对并集靶标做完整 GO/KEGG 富集
2. 再根据命令行传入的 `--disease-keywords` / `--disase-keywords` 构造关键词匹配规则
3. 用该规则去匹配富集结果中的 `Description`
4. 优先保留匹配到的 GO/KEGG 条目

例如传入：

```bash
--disease-keywords "NAFLD,liver,hepatic"
```

程序内部会按近似这样的规则做匹配：

```text
NAFLD|liver|hepatic
```

需要特别注意：

- 关键词不是去直接匹配原始靶标基因名
- 它们匹配的是富集结果表中的条目描述
- 匹配结果直接决定 `c` 图优先显示哪些条目
- 同时也会影响 `d` 图，因为 `d` 图的关键 KEGG 基因是从筛选后的 KEGG 条目中提取的
- 如果一个关键词都没匹配上，程序会回退到按显著性和 `Count` 排序的 top 条目，而不是直接输出空结果

输出到：

- `c_go_kegg/2.GO_result_2.xlsx`
- `c_go_kegg/2.KEGG_result_2.xlsx`
- `c_go_kegg/GO_KEGG_plot.xlsx`
- `c_go_kegg/2.GO_KEGG.pdf`

### Step D：KEGG circlize

从 `c` 图筛出的关键 KEGG 通路中提取关键基因，构建 `d` 图。

因此，`d` 图严格承接 `Step C` 的过滤结果，而不是直接从全部 KEGG 结果里取基因。

输出到：

- `d_kegg_circlize/key_targets_from_kegg.csv`
- `d_kegg_circlize/3.KEGG_circos.pdf`

### Step E：Chemprop

`e` 步必须严格从 `d` 图的关键 KEGG 基因继续往下做。

正确路径是：

1. `key_targets_from_kegg.csv`
2. 转成 `Chemprop` 候选靶标
3. 用 `ChEMBL` 人源活性数据构建训练表
4. 训练 `Chemprop`
5. 对当前单个 `CAS` 分子推理
6. 输出测试指标、过滤依据、过滤结果和 `Figure1e`

本步骤的主要产物包括：

- `e_chemprop/chemprop_data/`
- `e_chemprop/chemprop_model/`
- `e_chemprop/Figure1e_filtered.png`
- `e_chemprop/Figure1e_filtered.pdf`

## 命令行用法

### 运行完整分析

```bash
MIPTD \
  --cas 491-71-4 \
  --disase-keywords "NAFLD,liver,hepatic,steatosis"
```

常用参数：

- `--cas`
  输入分子的 `CAS`
- `--disease-keywords` / `--disase-keywords`
  疾病关键词，逗号分隔

这些关键词的用途是：

- 参与 `Step C` 的 GO/KEGG 条目描述过滤
- 间接决定 `Step D` 的关键 KEGG 基因集合
- 不直接用于过滤原始靶标基因表
- `--output-root`
  结果输出根目录，默认当前目录
- `--run-date`
  指定 case 目录日期后缀
- `--project-root`
  项目根目录
- `--dry-run`
  只打印命令，不实际执行

### 校验 case 目录

```bash
MIPTD-validate \
  --case-dir CAS_491-71-4_2026-03-21
```

校验内容包括：

- 五个步骤目录是否存在
- 关键结果文件是否存在
- `e` 步是否仍然是单一 `CAS`
- `prediction_summary.json` 是否只包含一个分子

## 失败策略

当前版本对在线抓取阶段采用“可降级但不假成功”的规则：

- `Swiss / SEA / ChEMBL / PPB2` 任一单源失败时，程序不会立刻终止
- 失败来源会被记录到 case 根目录的 `status.json`
- 对应来源会生成一个仅含表头的空占位文件，保证目录结构和下游读取格式稳定
- `Step A.5` 完成后，会检查四个来源标准化后的非空靶标集合数量
- 如果非空来源数不少于 `2`，流程继续执行 `b/c/d/e`
- 如果非空来源数少于 `2`，流程明确终止，并在 `status.json` 中标记失败

这样做的目的是：

- 允许像 `Swiss` 这种外部站点超时的情况被其余来源兜底
- 同时避免在证据严重不足时继续生成看似完整但实际上不可靠的结果

这个目录有两个强约束：

- 一个 `CAS` 只能对应一个 case 目录
- 一个 case 目录内只能保留这个 `CAS` 的结果
